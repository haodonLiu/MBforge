# MBForge 项目管理规范

> 版本：1.0 | 生效日期：2026-07-11

本规范定义需求、任务、里程碑、风险和技术决策的管理方式。目标是让工作可发现、
可排序、可验收、可追溯，而不是增加重复台账。

## 1. 信息源与职责

| 信息 | 单一来源 | 维护责任 |
|---|---|---|
| 产品入口与当前能力 | `README.md` | 维护者 |
| 优先级任务 | `TODO/INDEX.md` | 项目负责人 |
| 可讨论/可认领工作 | GitHub Issues | Issue 创建者与负责人 |
| 阶段目标 | `TODO/*-ROADMAP.md` | 里程碑负责人 |
| 实施方案 | `docs/superpowers/{specs,plans}/` | 功能负责人 |
| 长期架构决策 | `docs/adr/` | 技术负责人 |
| 开发与发布规则 | `CONTRIBUTING.md`、本文件、`docs/VERSION_CONTROL.md` | 维护者 |

`TODO/INDEX.md` 是优先级总表，Issue 是协作执行单元。两者并存时必须互相链接，禁止在
聊天记录或个人笔记中维护唯一任务状态。

## 2. 工作项类型与最小字段

类型使用 `bug`、`feature`、`refactor`、`test`、`docs`、`research`、`release`。
每个可执行工作项至少包含：

- 清晰的问题或目标，不把解决方案冒充需求；
- 范围内与范围外事项；
- 可验证的验收条件；
- 优先级、负责人、依赖和目标里程碑；
- 涉及数据迁移、兼容性、安全、模型成本时的风险说明。

超过 3 个模块、预计超过 3 个工作日、含不可逆迁移或改变公共契约的工作，应先写
spec/plan 并拆成可独立验收的子任务。

## 3. 优先级

| 级别 | 定义 | 响应规则 |
|---|---|---|
| P0 | 数据损坏、安全问题、主流程不可用、发布阻断 | 立即停止相关发布；优先修复 |
| P1 | 核心能力严重退化、结果可信度或关键体验受损 | 当前里程碑优先安排 |
| P2 | 一般缺陷、可维护性、性能和文档改进 | 按容量排期 |
| P3 | 低影响优化、探索和远期功能 | 进入候选池，定期复审 |

优先级由影响范围、严重程度、发生概率和时效性共同决定，不以请求者身份或提交时间
决定。P0/P1 必须写明影响与临时缓解措施。

## 4. 状态流转

```text
Backlog -> Ready -> In Progress -> Review -> Done
                 \-> Blocked -----/
```

- `Backlog`：已记录，尚未满足开工条件。
- `Ready`：范围、验收条件、依赖和负责人明确。
- `In Progress`：已有分支或正在实施；每人原则上同时只承担 1 个主要工作项。
- `Blocked`：外部依赖阻止进展，必须记录阻塞原因、责任方和复查日期。
- `Review`：实现完成，PR 已提交且验证证据齐全。
- `Done`：满足 `CONTRIBUTING.md` 的完成定义并已更新台账。

连续 14 天无进展的 `In Progress` 项应回到 `Ready` 或标为 `Blocked`；不保留名义上的
进行中任务。

## 5. 开工与完成门槛

Ready 条件：目标与价值明确；验收条件可测试；依赖已识别；数据/契约变更有方案；工作
规模适合一个 PR；负责人和目标里程碑明确。

Done 条件：代码或文档已合并；适用测试与质量门禁通过；用户文档、API 契约、迁移说明
和 `CHANGELOG.md` 已同步；Issue/TODO 状态已更新；遗留风险已新建工作项而非藏在 PR
评论中。

## 6. 计划与跟踪节奏

- 每周整理一次：核对 P0/P1、阻塞项、无负责人项和已失效描述。
- 每个里程碑开始时只承诺满足 Ready 的任务，并预留缺陷修复容量。
- 每个里程碑结束时核对验收结果、未完成项去向、质量指标和主要风险。
- 路线图表达目标与结果，不把尚未验证的日期写成承诺。

建议跟踪：P0/P1 未解决数量、任务周期时间、阻塞时间、回归缺陷数、测试通过率和关键
路径覆盖率。禁止用提交数或代码行数衡量个人绩效。

## 7. 决策、风险与变更控制

满足以下任一条件必须新增 ADR：改变模块边界或主要技术；改变持久化布局；引入难以
替换的基础依赖；改变兼容性或安全模型。ADR 使用连续编号，记录背景、决策、备选方案、
后果和状态；后续决策通过新 ADR 取代旧 ADR，不改写历史。

数据库、配置、工件路径和公共 API 变更必须说明：向后兼容窗口、迁移步骤、失败恢复、
回滚限制和验证证据。发现范围显著扩大时，暂停实现并更新工作项或拆分 PR。

## 8. 维护规则

规范变更本身也通过 PR 审查。规则与代码不一致时，先以可验证的当前行为保障运行，
随后在同一 PR 或明确关联的任务中修正文档；不得长期保留“口头例外”。每季度至少复审
一次本文件、贡献指南、版本规范和 Issue 模板。
